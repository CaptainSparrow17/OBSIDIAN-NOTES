
##### Nameserver
A nameserver is the Domain Name System (DNS) server component that translates a website’s domain name into a numerical IP address. This term can refer to any server with DNS software, as well as a hosting provider’s server that manages customer domain names.

##### DNS
The Domain Name System (DNS) protocol keeps records of which domain names correspond to specific IP addresses. DNS enables you to browse the web by typing in regular URLs instead of IP addresses.

##### Does DNS store anything?
❌ DNS itself = NO storage
DNS is just the system/rules.
But DNS uses servers that DO store data.

##### Where is the data actually stored?
The data (called **DNS records**) is stored on:
👉 **Name servers**
These are real servers that hold information like:
- `google.com → IP address`
- email servers (MX records)
- subdomains (`mail.google.com`)
So:
> Name servers = storage of DNS records

##### What exactly is a name server?
A **name server** is just a computer/server that:
- stores DNS records for a domain
- responds when asked “what is the IP for this domain?”
Think:
👉 It is like a database server for domain names.

DNS is mainly unencrypted. Devices on the local WLAN and Internet providers can therefore hack in and spy on DNS queries. Since this poses a privacy risk, there are now some solutions for DNS encryption. By default, IT security professionals apply `DNS over TLS` (`DoT`) or `DNS over HTTPS` (`DoH`) here. In addition, the network protocol `DNSCrypt` also encrypts the traffic between the computer and the name server.

However, the DNS does not only link computer names and IP addresses. It also stores and outputs additional information about the services associated with a domain. A DNS query can therefore also be used, for example, to determine which computer serves as the e-mail server for the domain in question or what the domain's name servers are called.
![](../../Attachements/Pasted%20image%2020260627210105.png)

Different `DNS records` are used for the DNS queries, which all have various tasks. Moreover, separate entries exist for different functions since we can set up mail servers and other servers for a domain.

|**DNS Record**|**Description**|
|---|---|
|`A`|Returns an IPv4 address of the requested domain as a result.|
|`AAAA`|Returns an IPv6 address of the requested domain.|
|`MX`|Returns the responsible mail servers as a result.|
|`NS`|Returns the DNS servers (nameservers) of the domain.|
|`TXT`|This record can contain various information. The all-rounder can be used, e.g., to validate the Google Search Console or validate SSL certificates. In addition, SPF and DMARC entries are set to validate mail traffic and protect it from spam.|
|`CNAME`|This record serves as an alias for another domain name. If you want the domain [www.hackthebox.eu](http://www.hackthebox.eu/) to point to the same IP as hackthebox.eu, you would create an A record for hackthebox.eu and a CNAME record for [www.hackthebox.eu](http://www.hackthebox.eu/).|
|`PTR`|The PTR record works the other way around (reverse lookup). It converts IP addresses into valid domain names.|
|`SOA`|Provides information about the corresponding DNS zone and email address of the administrative contact.|
##### SOA RECORDS
```shell
dig soa www.inlanefreight.com
```
The `SOA` record is located in a domain's zone file and specifies who is responsible for the operation of the domain and how DNS information for the domain is managed.
The dot (.) is replaced by an at sign (@) in the email address. In this example, the email address of the administrator is `awsdns-hostmaster@amazon.com`.

### Default Configuration
There are many different configuration types for DNS. Therefore, we will only discuss the most important ones to illustrate better the functional principle from an administrative point of view. All DNS servers work with three different types of configuration files:

1. local DNS configuration files
2. zone files
3. reverse name resolution files

The DNS server [Bind9](https://www.isc.org/bind/) is very often used on Linux-based distributions. Its local configuration file (`named.conf`) is roughly divided into two sections, firstly the options section for general settings and secondly the zone entries for the individual domains. The local configuration files are usually:

- `named.conf.local`
- `named.conf.options`
- `named.conf.log`

It contains the associated RFC where we can customize the server to our needs and our domain structure with the individual zones for different domains. The configuration file `named.conf` is divided into several options that control the behavior of the name server. A distinction is made between `global options` and `zone options`.

Global options are general and affect all zones. A zone option only affects the zone to which it is assigned. Options not listed in named.conf have default values. If an option is both global and zone-specific, then the zone option takes precedence.

##### DIG-NS Query
The footprinting at DNS servers is done as a result of the requests we send. So, first of all, the DNS server can be queried as to which other name servers are known. We do this using the NS record and the specification of the DNS server we want to query using the `@` character. This is because if there are other DNS servers, we can also use them and query the records. However, other DNS servers may be configured differently and, in addition, may be permanent for other zones.

```shell
dig ns inlanefreight.htb @10.129.14.128
```
![](../../Attachements/Pasted%20image%2020260627212933.png)


##### DIG-Version Query
Sometimes it is also possible to query a DNS server's version using a class CHAOS query and type TXT. However, this entry must exist on the DNS server. For this, we could use the following command:
```shell
dig CH TXT version.bind 10.129.120.85
```
![](../../Attachements/Pasted%20image%2020260627213028.png)

##### DIG - ANY Query
We can use the option `ANY` to view all available records. This will cause the server to show us all available entries that it is willing to disclose. It is important to note that not all entries from the zones will be shown.

```shell
dig any inlanefreight.htb @10.129.14.128
```

###### Canonical name
A CNAME (Canonical Name) record is a Domain Name System (DNS) setting that acts like an alias, forwarding one domain or subdomain to another hostname instead of a direct IP address.
It allows multiple domain names to resolve to the same IP address and simplifies the process of managing various services or subdomains by pointing them to a single destination domain. Notably, all CNAME records must direct to a domain name rather than an IP address. The ”www” subdomain is a common CNAME example.

#### DNS ZONE TRANSFER
A DNS zone transfer (AXFR/IXFR) is the process of replicating DNS records from a primary DNS server to a secondary server. It ensures consistent records across the authoritative infrastructure for redundancy and fault tolerance. Operating over TCP port 53, it can be a significant security vulnerability if left improperly configured.

To answer this in a bit more context, a name server is a server that is "authoritative" (is authorized to respond to DNS requests) for your domain.

For most people this will be your name servers for your hosting company ns1.somehostingcompany.com and ns2.somehostingcompany.com.

For "custom" name servers, what the hosting company is talking about is creating name servers under your domain. So instead of the ones for the hosting company, the domain will instead point to ns1.yourdomain.com and ns2.yourdomain.com.

In reality there is no difference between the two, and it's more of a cosmetic thing than anything else. However if you were running your own hosting company and reselling somebody else's services, then you might want to hide that from your customers.
